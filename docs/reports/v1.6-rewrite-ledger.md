# Fusekit v1.6 rewrite evidence ledger

This Phase 1 audit is dated 2026-07-19. It is the evidence and disposition ledger
for the hard-cut rewrite. It records why each subsystem was retained, moved, or
deleted and its final implementation disposition.

## Decision

Fusekit v1.6 is a clean protocol and architecture break. It does not preserve
old APIs, wire versions, journals, migration paths, or mixed-version behavior.
All direct consumers move manually in the same release train. There is no
canary or compatibility period.

The rewrite has three ownership boundaries:

- daemonkit owns generic daemon identity, signed-peer trust, process takeover,
  crash-safe generation draining, child supervision, and bounded wire serving;
- Fusekit owns filesystem identity, content revisions, materialization,
  enumeration deltas, mutation convergence, mounts, leases, and teardown;
- consumers own domain policy, content production, and their signed app
  identity and packaging. In particular, cc-pool owns Claude-specific path
  classification and settings synthesis, but not File Provider identity,
  invalidation, retry, or lifecycle mechanics.

This split is the gate for every design decision below. A mechanism needed by
two filesystem consumers belongs in Fusekit. A mechanism with no filesystem
semantics belongs in daemonkit. Consumer-side workarounds are deleted, not
renamed.

Two historical results remain non-negotiable:

1. no code path force-unmounts without the full carcass proof and an exclusive

   Neither may depend on a Cellar path, ad-hoc binary, copied executable, or
   temporary or versioned application path.

## Current architecture delta

### Deleted Fusekit mechanisms

- `mountd`, its protocol envelopes, holder server, journal replay, retirement,
  client/spawn surface, and feature negotiation are gone.
- The root in-process mount lifecycle, `MountSet`, `MountSpec`, mount-table
  probes, old live tests, and standalone holder binary are gone.
- `content.Source`, `Tree`, `WritableTree`, `HandleTree`, their bridge and relay,
  `holderfs`, overlay selection, and the old File Provider control runtime are
  gone.
- Per-directory lease state, process strikes, retirement breakers, versioned
  holder state, the VM stress driver, and its legacy scenario modes are gone.
- The only retained native primitives are the signed child's FUSE-T facts and
  generic AppleDouble blocking policy.

CatalogFS parity is pinned directly: opaque object/inode identity, exact
generation fences, immutable paged enumeration, open snapshots surviving
replacement, prepared mutations, transactional replace-over-target, same-tenant
serialization, and AppleDouble read/write behavior all have focused tests.

### Composed daemonkit mechanisms

The rewrite composes daemonkit takeover, drain, trust, exact persistent
sessions, process identity, reaping, and disposable workers through
`daemon.Runtime`. FuseKit adds filesystem messages and semantic scheduling but
does not wrap those mechanisms in near-duplicates. daemonkit owns listener and
shutdown ordering; FuseKit owns catalog and tenant convergence.

Before release, daemonkit must publish the exact composition seams already used
by the holder and cut a minor release. The corresponding release is `v0.2.0`;
FuseKit does not retain local copies while waiting for the tag.

## File Provider amplification ledger

The initiating `.claude.json` or `settings.json` write is real but is not the
failure. The current stack amplifies one semantic change into multi-domain
reconciliation and then extends it through document-ID churn.

| Link | Current behavior | Required invariant |
|---|---|---|
| Event fan-out | cc-pool's coordinator loops over active accounts and signals every domain, even when a domain's effective revision did not change. | Fusekit computes an effective revision per domain and emits one latest-wins invalidation only to domains whose revision advanced. |
| Temporary write | The extension creates a private temporary item during an atomic write. | Temporary names and final names are aliases in one mutation transaction, not independent identities. |
| Promotion | Mutation policy removes the private temporary item and returns a computed item whose role-derived `ItemID` differs. | One object keeps its opaque File Provider identifier across private, materializing, computed, renamed, and refreshed states. Reusing a deleted path creates a new object identifier. |
| Enumeration | Each retry rebuilds the complete root and performs bridge reads. | Enumeration is revisioned and delta-based. Unchanged nodes require no content read and preserve document identifiers. |
| Retry | `itemCollision`, `ESTALE`, missing-on-disk cleanup, and delayed continuation cause macOS to retry and relaunch extensions. | Collision and stale-ID conditions are prevented by construction; retries are bounded to a revision and coalesced. |

The cc-pool migration must delete its global notify loop and role-derived
identity policy. cc-pool provides domain content and classification through the
new Fusekit API. It must not retain an alternate File Provider coordinator,
identity table, enumeration cache, or retry loop.

Acceptance for the reported incident is stronger than "no crash." One base
configuration write produces at most one effective-revision advance per
affected domain, zero `itemCollision` events, zero wrong-document `ESTALE`
events, no full-root rebuild for unchanged entries, and no repeated extension
launch burst after convergence.

### Explicitly reject

- the compatibility proxy and mixed-generation wire design in
  `now-that-we-have-golden-bunny.md`; daemonkit performs a hard generation
  transition, and every consumer upgrades together;
- stable executable copying, including `Spawn.RefreshStable`, stable consumer
  directories, or resurrecting `ReexecStable`; signed installed apps are the
  only executable identity;
- direct daemon reads through File Provider domains, widget-only freshness
  protocols, or cc-pool-owned enumeration and identity recovery;
- longer operation deadlines as a readiness fix. Deadlines bound work but do
  not establish identity, freshness, or materialization;
- automatic fallback to symlink or another backend after a contract failure.
  Selection may choose an explicitly configured healthy backend, but it does
  not hide corruption by switching implementations;
- mass-kill watchdogs, periodic domain bouncing, and retry loops that do not
  carry an expected revision and a bounded attempt budget.

## Immediate recovery baseline

The Phase 0 recovery on `yasyf-home` restarted cc-pool v0.59.0 only through
Homebrew. The old daemon exited cleanly, repeated account-18 catch-up completed
in under 150 ms, the supported `CCP_ACCOUNT=18 ccp run --version` path exited in
140 ms, and the final state had no account lock holder, lease holder, active
session, deferred apply, or deadline error.

The raw `ccp select` SSH probe exposed another ownership bug: session-leader
discovery associated the lease agent with the root `tailscaled` process, so the
lease could not expire with the SSH command. The exact test lease agent also
ignored TERM and required identity-gated KILL. The rewrite must bind a session
lease to a consumer identity that proves direct liveness and ownership; an
unrelated ancestor or session leader is invalid. Lease termination must use a
bounded TERM/KILL/reap ladder, and restart recovery must discard advisory slots
whose recorded child identity no longer exists.

## Release and migration ledger

The release train is ordered and intentionally uncompromising.

1. daemonkit v0.2.0 publishes the final holder composition seams;
2. Fusekit v1.6.0 publishes the single new protocol and Go/Swift libraries;
3. cc-pool v0.60.0, cc-notes v0.36.0, and cc-squash v0.2.0 manually adopt it;
4. each production host upgrades in one hard cut by stopping admission,
   draining and acknowledging durable writes, stopping old services,
   unmounting the old holder, removing old File Provider domains while old
   tooling can still identify them, and archiving derived sockets and journals
   for forensics;
5. install only the complete new generation, rebuild desired tenant state from
   consumer source-of-truth data, and resume admission after end-to-end checks.

There are no canaries, dual readers, old-protocol decoders, feature probes for
mixed fleets, or rollback shims. A failure blocks the train and is fixed
forward.

The 2026-07-19 live dependency sweep expands the migration set beyond the
three primary Fusekit consumers. The direct Fusekit API consumers are cc-pool,
cc-notes, and cc-squash. Daemonkit API or wire consumers include Fusekit,
cc-pool, synckit, cc-interact, cc-orchestrate, and captain-hook's generated
Swift project. The cc-skills repository supplies generated Go daemon
templates. The transitive verification and repin set includes cc-present and
cc-runtime. The shared Homebrew tap remains release metadata regenerated by
its owning workflow.

This inventory is evidence from the current checkout, not an allowlist. The
same repository and template sweep runs again immediately before tags are cut.

## Verification gates

The implementation is complete only when all of these hold:

- every repository uses its test wrapper, never bare `go test`; the FuseKit
  gate covers pure, race, fuse-tag compilation, Swift, and isolated-VM suites;
- protocol golden tests prove one schema across Go and Swift, with regeneration
  required to be a no-op;
- crash injection at each drain transition proves single ownership and exact
  worker settlement; no legacy carcass or force path exists;
- signed-peer tests reject wrong Team IDs, identifiers, unsigned peers,
  injection-permissive peers, and stale process identity;
- File Provider mutation tests prove stable IDs through atomic write, rename,
  private-to-computed transition, delete/recreate, and materialization;
- delta enumeration and coalescing tests prove unchanged roots trigger no
  bridge reads and a burst converges to the latest revision;
- the signed-child VM suite proves stable attributes, blocked AppleDouble
  behavior, exact child reaping, and no force-unmount path;
- TCC validation across two consecutive upgrades produces no new consent row or
  prompt for the holder or cc-pool daemon identities;
- the yasyf-home `acct-18 content catch-up: context deadline exceeded` path is
  reproduced before the cut and eliminated after it without increasing a
  timeout. Selection observes typed claim ownership and content revision, waits
  only on a cancellable operation, and either completes or returns the precise
  blocking owner/state;
- the reported 14-domain File Provider workload produces one bounded
  invalidation per changed domain, stable document IDs, zero collision/ESTALE
  amplification, and no delayed-continuation launch storm.

History and plan classifications are evidence, not compatibility commitments.
If an old behavior cannot be expressed by the new ownership, identity,
revision, and generation contracts, it is deleted.
